## 📊 快速参考

### 核心概念速查

| 概念 | 定义 | 解决方案 |
|-----|------|---------|
| **告警风暴** | 短时间内产生大量告警 | 告警关联 + 分组聚合 |
| **根因分析** | 识别问题根本原因 | 依赖关系图 + 时间窗口关联 |
| **告警噪音** | 无效或低价值告警 | 动态阈值 + 多数据点评估 |
| **告警抖动** | 频繁触发和恢复 | 滞后机制 + 评估周期优化 |
| **告警疲劳** | 因告警过多导致麻木 | 优先级分级 + 噪音过滤 |

### 技术栈对比速查

| 维度 | CloudWatch | Prometheus + Grafana |
|-----|-----------|---------------------|
| **成本** | 按使用量付费 | 开源免费（需自建） |
| **集成** | AWS 原生集成 | 需要额外配置 |
| **灵活性** | 中等 | 高 |
| **学习曲线** | 低 | 中等 |
| **社区支持** | AWS 官方 | 开源社区 |
| **推荐场景** | AWS 环境 | 混合云/多云 |

### 关键指标速查

| 指标 | 目标值 | 说明 |
|-----|-------|------|
| 告警准确率 | > 80% | 真实问题告警占比 |
| 告警噪音率 | < 20% | 误报和重复告警占比 |
| 告警抖动率 | < 10% | 频繁触发恢复的告警占比 |
| 平均 MTTR | < 15 分钟 | 问题修复时间 |
| 告警响应时间 | < 5 分钟 | 告警确认时间 |
| 告警风暴频率 | < 1 次/月 | 单次事件 > 50 条告警 |

## 💡 最佳实践

### ✅ 推荐做法

1. **分阶段实施**: 先快速止血，再深度优化
2. **保留备份**: 实施新方案时保留原有告警作为备份
3. **充分测试**: 在测试环境验证后再上生产
4. **团队培训**: 确保团队理解新的告警机制
5. **持续优化**: 定期审查和调整告警规则

### ❌ 避免的错误

1. **过度抑制**: 不要为了减少告警而过度抑制
2. **一刀切**: 不同服务需要不同的告警策略
3. **忽视监控**: 要监控告警系统本身的健康度
4. **缺少文档**: 要记录告警规则的业务含义
5. **孤立实施**: 要与团队充分沟通和协作
